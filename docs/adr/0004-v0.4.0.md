# ADR 0004 — v0.4.0: per-dot PPU, save states, and the mid-catalog mappers

- **Status:** Accepted
- **Release:** v0.4.0 (2026-05-24)
- **Theme:** Turn the PPU into a per-dot state machine and ship the persistence + mappers that unlock the bulk of the licensed library.

## D1 — Per-dot `stepDot` PPU model

- **Context:** the prior PPU advanced in coarse per-scanline batches, which made the sprite-0 hit land at frame-start (one big render) and made the odd-frame dot-skip impossible to phase against a `$2001` write at a specific dot. Both are timing-visible: SMB1 desyncs its scroll without the dot-skip, and any status-bar split needs sprite-0 to latch at the real scanline of the overlap.
- **Decision:** the PPU advances one PPU dot at a time inside a master-clock deadline. `Run(masterClockDeadline)` loops `stepDot()` while `masterClock + PPUMasterClockDividerNTSC <= deadline`, advancing `masterClock` by one PPU-dot's worth of master clocks per dot. Each `stepDot` increments `dot`, wraps at `DotsPerScanline` (bumping `scanline`/`frameCount`), and runs the per-dot vblank-set/clear, per-scanline BG render, and per-scanline sprite-0 detector inline rather than at frame boundaries.
- **Consequences:** sprite-0 hit latches `$2002` bit 6 at the actual scanline of the overlap; the odd-frame dot-skip becomes a real per-dot decision (see D2). This per-dot deadline loop is the spine every later accuracy refinement hangs off — the MMC3 A12-via-PPUADDR clocking (ADR 0008) and the deferred `$2006` v-update both reason in terms of the dot the PPU is on, which only exists because of this model.

## D2 — Odd-frame dot-skip via a delayed rendering-enabled latch

- **Context:** on NTSC with rendering on, the pre-render scanline is one dot shorter on odd frames (the PPU jumps from dot 339 straight to (0,0)). The skip decision must sample rendering-enabled at dot 339, and Mesen2 models that sample with a 1-PPU-clock delay relative to the live mask. Sampling the live `$2001` mask directly mis-times the skip against a BG-enable write near dot 339.
- **Decision:** keep `renderingEnabledDelayed`, synced at the end of each `stepDot` from the live mask (`mask & 0x18 != 0`), and latch `oddSkipArmed = renderingEnabledDelayed` at pre-render dot 339. The boundary check at the next dot shortens the scanline by one only when `OddFrameSkip && scanline == PreRenderScanline && oddSkipArmed && frameCount is odd`.
- **Consequences:** this is the load-bearing "rendering-enabled has a 1-PPU-clock delay" invariant recorded in `CLAUDE.md`. It is what eventually closed the last `ppu_vbl_nmi` sub-test once the master-clock interleave landed in the chippy core; the latch itself is established here at the per-dot layer.

## D3 — Save-state v1 schema: gob + gzip, keyed by ROM SHA-256

- **Context:** want quick-save / quick-load slots that round-trip the whole machine (CPU, RAM, PPU, APU, joypad, cart) and survive across runs, without a hand-rolled binary format to version by hand.
- **Decision:** a top-level `nesSaveState` envelope (`Magic = "NESSAVE\x00"`, `Version = 1`, `ROMHash`) wrapping each subsystem's `FullState`, gob-encoded inside a gzip wrapper (NES framebuffers compress 10x+). Slot files live at `~/.nessy/states/<rom-hash[:16]>-slot<N>.state`, keyed by the SHA-256 of the ROM so renaming a ROM never orphans its saves. Load fails fast on a magic or version mismatch and refuses a save captured for a different ROM (`ROMHash` check). Capture/apply run under `cpuMu` so state is read/written at an instruction boundary; restore applies the CPU last so cart-restore IRQ re-assertions don't clobber it.
- **Consequences:** the format is **backward-compatible within the major version** — new fields stay optional inside v1.x; a semantic change or field removal requires bumping `stateVersion` plus a migration. This freeze contract is what later DAP-driven and headliner-playtest work (the v1.0 epic) relies on so old slot files keep loading.

## D4 — Battery-backed PRG-RAM persistence (.sav)

- **Context:** mappers with battery-backed PRG-RAM (Zelda-class saves, MMC3 RPGs) must persist `$6000-$7FFF` between sessions, separate from full save-state slots.
- **Decision:** a separate `.sav` path under `~/.nessy/sav/<sha256-of-rom>.sav`, again keyed by ROM hash. The cart exposes `BatteryBacked() bool` + `PRGRAM() []byte`; `loadBattery` copies the file into PRG-RAM on boot when the cart is battery-backed and the sizes match, `saveBattery` writes it back. Both are best-effort: a missing or mismatched `.sav` logs to stderr and is ignored, never fatal — a user who can't load a save can still play.
- **Consequences:** battery saves are decoupled from save-state slots (a different directory, a different keying width, a raw `[]byte` rather than the gob envelope), so the two persistence stories version independently. Carts that ship no PRG-RAM (UxROM, CNROM) return `false`/`nil` and are skipped for free.

## D5 — UxROM (mapper 2) and CNROM (mapper 3)

- **Context:** the early licensed catalog (Mega Man 1/2, Castlevania, Contra, DuckTales, Metal Gear via UxROM; Arkanoid, Bump'n'Jump and the sport titles via CNROM) needs the two simplest bank-switchers beyond NROM/MMC1.
- **Decision:** `UxROM` switches a 16 KiB PRG bank at `$8000-$BFFF` from the low nibble of any `$8000-$FFFF` write, hardwires the last bank at `$C000-$FFFF`, and ships 8 KiB CHR-RAM. `CNROM` keeps a fixed NROM-style PRG window and switches an 8 KiB CHR bank from the write data. Both pin mirroring at construction from the iNES header and modulo the bank index by the real bank count so over-wide writes wrap. The UNROM bus-conflict quirk (AND the written byte with the ROM byte) is gated behind iNES 2.0 sub-mapper 2 rather than applied unconditionally.
- **Consequences:** both implement the same `cart.Cartridge` surface (including the `BatteryBacked`/`PRGRAM` pair from D4, returning the empty answer), so the bus wiring and persistence paths treat every mapper uniformly. Bus-conflict emulation is opt-in per sub-mapper to avoid breaking the common ROMs that don't exercise it.

## D6 — MMC3 (mapper 4) with the A12 scanline IRQ

- **Context:** MMC3 is the late-NES workhorse (SMB3, Mega Man 3-6, Kirby's Adventure, Crystalis, Battletoads). Its headline feature is the scanline IRQ that fires on A12 rising edges so games split the screen by fetch timing instead of sprite-0 polling.
- **Decision:** the full register file (`$8000/$8001` bank select+data, `$A000` mirroring, `$A001` PRG-RAM protect, `$C000/$C001` IRQ latch+reload, `$E000/$E001` IRQ disable+ack / enable) with PRG modes 0/1 and CHR A12-invert modes. The A12 counter clocks on the rising edge of PPU-bus bit 12 (`clockA12`), debounced by `prevA12` so a low dwell is required before the next edge counts; on counter underflow with IRQ enabled it asserts via the named-source IRQ sink (`"mmc3"`), and `$E000` acks. Both the Sharp **RevB** (default) and NEC **RevA** (sub-mapper 3, Klax) reload/fire orderings are implemented.
- **Consequences:** MMC3 is the first mapper to drive the CPU IRQ line, so it establishes the `IRQSink` named-source contract (`AssertIRQSource`/`ClearIRQSource`) that later expansion-IRQ mappers reuse. At v0.4.0 the counter is clocked purely through CHR fetches in `PPURead`/`PPUWrite`. That under-counts edges that the PPU drives via `$2006`/`$2007` while rendering is off — the `NotifyVRAMAddr` PPUADDR hook (sharing `prevA12` so a single rise can't double-count) is the **ADR 0008-era** refinement that closes the remaining `mmc3_test` sub-tests; the per-dot model from D1 is what makes that hook's "gate on `!renderingEnabled()`" reasoning possible.
