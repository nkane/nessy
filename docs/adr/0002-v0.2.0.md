# ADR 0002 — v0.2.0: per-scanline sprites, OAMDMA bus-steal, mid-frame scroll

- **Status:** Accepted
- **Release:** v0.2.0 (2026-05-20; folds v0.2.1)
- **Theme:** Move the 2C02 from a per-frame blitter to a per-scanline renderer with sprites and mid-frame scroll, and steal the bus for `$4014` OAMDMA — enough to boot Super Mario Bros. 1.

## D1 — `$4014` OAMDMA via a CPU bus-steal hook

- **Context:** sprites need their 256-byte OAM populated, and games drive that through a single `$4014` write that DMAs a CPU page into OAM while stalling the CPU ~513 cycles. The DMA must consume real CPU cycles so the PPU (ticked per CPU cycle, ADR 0001) advances the right amount during the steal.
- **Decision:** ship an `internal/nes/dma` package whose `OAMDMA` peripheral claims `$4014`; on write of byte `$XX` it hands the source page to the CPU, which owns the cycle-by-cycle bus-steal. The original v0.2.0 cut used the chippy `cpu.Stall(n)` API — queue 513 bus-steal cycles, drained as one block on the CPU's next `Step` with the bus ticker firing once over the full delta, and NMI/IRQ service preempting a queued stall at the next instruction boundary.
- **Consequences:** sprites have data and the DMA cost is visible to the PPU. The `cpu.Stall` block model was deliberately coarse — it skipped the 514-cycle odd-CPU-cycle alignment penalty and per-byte sub-cycle accounting (no DMC contention to interleave with yet). That seam was later replaced by Mesen2's `ProcessPendingDma` model: `OAMDMA.Write` now flags `cpu.SetNeedSpriteDma(page)` and the CPU drains the 513/514-cycle window inside its read path at the correct cycle parity (chippy#377, see ADR 0004). The old `OAMDMA.Step`/external-drive path is gone — the state-signal model is the only path.

## D2 — Per-scanline rendering replaces the per-frame blit

- **Context:** the v0.1 PPU (ADR 0001) rasterized the whole frame at vblank entry from one scroll/state snapshot. That cannot express any mid-frame effect — and SMB1's static status bar over a scrolling playfield is exactly a mid-frame change.
- **Decision:** refactor `renderFrame` to walk scanlines 0..239 and render each row independently from a `scrollSnapshot{scanline, scrollX, scrollY, baseNametable}`. A single-tile-row cache keeps the common per-pixel case at one bus read.
- **Consequences:** the renderer is now a per-row pipeline, which is the structural prerequisite for D3 (mid-frame scroll) and D4 (sprite-0 hit) — and for every later raster-timed effect. It is still a scanline-granular model, not the per-dot fetch pipeline; cycle-exact A12 / sprite-fetch timing came much later.

## D3 — Mid-frame scroll via a per-scanline `$2005`/`$2006`/`$2000` event log

- **Context:** with per-scanline rendering in place, scroll writes that land mid-frame must affect only the rows that follow. SMB1 splits the screen by rewriting scroll during rendering.
- **Decision:** `stepDot` snapshots `frameStartScroll` when the scanline rolls back to 0 (capturing whatever the game wrote during vblank); any scroll-relevant `$2000`/`$2005`/`$2006` write during scanlines 0..239 appends a `scrollSnapshot` to a per-frame `scrollEvents` log that `renderFrame` advances through row by row. `scrollFromV()` synthesizes `scrollX`/`scrollY`/`baseNametable` from the 15-bit loopy `v` latch so `$2006` mid-frame address pairs (SMB1's split mechanism) produce correct snapshots, with horizontal + vertical nametable wrap (PPUCTRL bits 0/1) at the 256/240 boundaries.
- **Consequences:** mid-frame scroll splits work at scanline granularity. The event-log approach is a faithful-enough approximation of the loopy register dataflow without modelling the per-dot `v`/`t` copies; sub-cycle render-time `v` updates stayed out of scope.

## D4 — Sprite layer, sprite-0 hit, and sprite-overflow groundwork

- **Context:** booting SMB1 needs sprites composited over the background, and the title-to-game handoff depends on the sprite-0 hit flag (`$2002` bit 6) for its raster split timing.
- **Decision:** `renderFrame` seeds a `bgOpaque` mask alongside the framebuffer; `renderSprites` composites the 64-sprite OAM on top (lower OAM index wins on overlap), honoring attribute palette, behind-BG priority, h/v flip, and 8x8 / 8x16 modes (8x16 routes the pattern table via tile-index bit 0, ignoring PPUCTRL bit 3). Sprite-0 hit latches whenever an opaque sprite-0 pixel coincides with an opaque BG pixel, gated by both PPUMASK show-BG and show-sprites bits. Sprite overflow (`$2002` bit 5) sets when a scanline crosses more than 8 sprites.
- **Consequences:** SMB1 renders and its status-bar split fires. Both sprite-0 hit and overflow are the "simple-correct" model — overflow is the >8-sprites rule, NOT the 2C02 sprite-overflow silicon bug, and hit is frame-granular rather than dot-exact. These are explicitly groundwork: cycle-accurate sprite-0 hit + the overflow hardware bug were deferred to the later accuracy-ROM work.

## D5 — SMB1 boot as the milestone gate

- **Context:** the v0.1 core (ADR 0001) ran test ROMs and demos but no commercial game end-to-end. A real-game boot is the integration proof that OAMDMA, per-scanline rendering, scroll, and sprites compose correctly.
- **Decision:** treat "SMB1 boots, scrolls, and shows the split status bar" as the release's acceptance bar, on top of keeping the existing nestest / Klaus / decimal suites green with no perfgate regression.
- **Consequences:** a concrete, observable target that exercises every D1–D4 path at once. It also exposed the accuracy gaps those features punted on (DMA alignment, dot-exact sprite-0, the overflow bug), which became the backlog the later accuracy-ROM epic worked through. The release also folded in the first APU sound output (pulse 1/2 + frame counter, Ebiten audio sink), which the v0.3 audio work then built out — captured in ADR 0003.
