# ADR 0008 — v0.8.0: chippy carve-out + Mesen2 as the cycle-precision reference

- **Status:** Accepted
- **Release:** v0.8.0 (2026-05-25)
- **Theme:** Split nessy out of the chippy monorepo into its own repo, stand up the CI + release foundation, and adopt Mesen2 as the single source of truth for every cycle-precision invariant.

## D1 — Carve nessy out of the chippy monorepo into its own repo

- **Context:** chippy is library-shaped — a 6502 CPU + bus + DAP server with no graphics dependencies. nessy needs Ebiten, which drags in CGO and X11/GL on Linux. While both lived in one repo (the monorepo decision of `ADR 0002` D5), anyone consuming chippy's CPU as a library had to pull the entire NES + Ebiten dependency graph. The two sides had also diverged in release cadence: chippy ships clean 6502 library tags, nessy ships emulator binaries.
- **Decision:** execute the repo split tracked under chippy#386. nessy becomes its own repository depending on `github.com/nkane/chippy` as a normal pinned module (`go.mod`), consuming the public `cpu`, `dap`, `peripheral`, `symbols`, `trace`, `loader`, and `expr` packages. The chippy-side accuracy + sub-cycle work that landed across many PRs while the two were a single repo (the decisions captured in `ADR 0001`–`ADR 0007`) therefore becomes a **library dependency** rather than in-tree code — most of the load-bearing CPU invariants below now live in the dep, not this repo.
- **Consequences:** each side versions on its own cadence; chippy ships as a clean 6502 library with no GL deps. The cost is a two-repo dependency edge: a CPU-core fix now requires a chippy release + a `go.mod` bump here. chippy's public API is semver-stable, so bumping is safe within a major; a chippy major bump requires matching nessy updates. The carve preserved git history via `git filter-repo`. This realizes the "split later" half of `ADR 0002` D5's deliberately deferred three-way-split call.

## D2 — Rename release tags `nessy-v:v`

- **Context:** inside the monorepo, nessy releases were tagged `nessy-v*` to disambiguate from chippy's bare `vX.Y.Z` tags sharing the same git history. After the carve there is no chippy tag to collide with — the repo is nessy's alone — and the `nessy-v` prefix is now noise that breaks goreleaser's default `vX.Y.Z` expectations and the `go install module@vX.Y.Z` convention.
- **Decision:** rename every historical tag with `git filter-repo --tag-rename nessy-v:v` so nessy ships under bare `vX.Y.Z` tags. v0.8.0 is the last release cut around the carve; v0.9.x / v1.0.0 follow under the v1.0 epic.
- **Consequences:** tags now match Go module semantics and goreleaser's defaults, clearing the path for the release-pipeline modernisation (cosign + Homebrew tap + AUR) deferred to a later milestone. Anyone with the old `nessy-v*` tags fetched locally must re-fetch; the rename rewrote tag refs.

## D3 — CI + release foundation

- **Context:** post-carve nessy needs its own quality gate — the monorepo's shared CI no longer covers it — and a reproducible way to run the GPL test ROMs that gate accuracy.
- **Decision:** stand up a single `.github/workflows/ci.yml` running the quality bar on every PR: `go build ./...` (and the full `-tags=nessy` build for the Ebiten binaries), `go test -race`, lint, and the accuracy job. GPL test ROMs are never vendored — the accuracy CI job downloads them on demand, sha256-pins them, and caches the downloads with retry-on-transient-failure. The Ebiten command binaries stay behind the `//go:build nessy` tag inherited from the monorepo (where it isolated Ebiten from chippy's default build); stripping it post-carve would churn ~100 files for no functional benefit, so it stays.
- **Consequences:** PRs get a green/red signal independent of chippy. Default `go build ./...` stays fast and GL-free; the full emulator build is the explicit `-tags=nessy` invocation. The accuracy harness (`cmd/nessy/accuracy_test.go`, `accuracyROMs` slice) is the regression safety net — a real regression in a passing ROM fails CI, while tracked gaps carry a `knownFail` string and skip.

## D4 — Adopt Mesen2 as the cycle-precision reference

- **Context:** "cycle-accurate" needs an oracle. nesdev wiki prose is necessary but underspecifies sub-cycle ordering (interrupt poll timing, DMA bus-steal parity, PPU render-enable latency). Independent guesswork produced ROMs that passed one sub-test and regressed another.
- **Decision:** make Mesen2's `Core/NES/` the single source of truth for every cycle-precision detail. When a behavior is ambiguous, the implementation matches Mesen2's, and every load-bearing borrow carries an explicit Mesen source line reference in the PR thread and in `CLAUDE.md`. The per-cycle interleave path is gated on `cpu.VariantNES` — chippy keeps its instruction-stepped batch tick for NMOS / 65C02 so the chippy debugger, Klaus functional tests, and BCD sweeps stay byte-identical. This refines the per-cycle interleave first introduced in `ADR 0004`.
- **Consequences:** a consistent oracle ended the whack-a-mole. The borrows below all trace to specific Mesen files. The cost is a hard dependency on one reference emulator's interpretation; where Mesen models a quirk nessy hasn't ported yet (e.g. the deferred $2006 v-update, 3 PPU cycles), the affected sub-tests are tracked-skipped rather than approximated.

## D5 — Master-clock model + per-cycle CPU↔PPU interleave

- **Context:** the PPU and cart peripherals must advance in lockstep with the CPU mid-cycle, not batched at instruction boundaries, or interrupt-timing and sprite-0 ROMs fail.
- **Decision (chippy-side, in the dependency):** for `VariantNES`, `cpu.Step` runs 1:1 — every bus access ticks the chain one cycle, with addressing-mode dummy reads added per template (`c.idle`, `addrDummies`). The master-clock model (chippy#375) uses NTSC 12 mc/CPU cycle, 4 mc/PPU dot, `ppuOffset=1`; a read splits its master-clock budget +5 pre / +7 post around the bus access, a write +7 pre / +5 post, calling `PPU.Run(deadline)` at each split (mirrors Mesen2 `Start/EndCpuCycle`). The `instrCycles == accounted` assertion at end of each instruction is a proven invariant guard — if it fires, a dummy-cycle template is wrong.
- **Consequences:** the PPU catches up to the CPU's exact mid-cycle phase, dot by dot. This is the substrate every PPU- and DMA-timing ROM depends on, and refines `ADR 0004`'s per-cycle interleave with the explicit master-clock budget.

## D6 — Interrupt sub-cycle ordering: NMI hijack, poll-latch delay, branch quirk

- **Context:** the `cpu_interrupts_v2` suite pins ordering details a naïve interrupt path gets wrong.
- **Decision (chippy-side):** the **NMI hijack check sits between `push16(PC)` and `push(P)`** in `serviceVector` (Mesen2 `NesCpu::IRQ` order) — this cleared `nmi_and_irq` (test 3). **NMI/IRQ poll latches sample at the penultimate cycle** via `nmiPollPrev` / `irqPollPrev` one-cycle delay — `cli_latency` and `nmi_and_brk` depend on this. The **branch IRQ-poll quirk** (chippy#377): a taken non-page-cross branch ignores an IRQ asserted at its last clock; `branch()` rolls back `irqPollPrev` if it just rose this cycle (Mesen `NesCpu::BranchRelative`). The `serviceNMI` edge latch is cleared BEFORE the 7 service cycles or a spurious second NMI fires.
- **Consequences:** `cpu_interrupts_v2` reaches 5/5. These are silent, high-leverage ordering rules — getting any one wrong breaks a different sub-test, which is exactly why D4's single oracle mattered.

## D7 — Mesen2 `ProcessPendingDma` port (OAMDMA + DMC bus-steal)

- **Context:** OAMDMA ($4014) and DMC sample fetches both steal the bus, and they contend with each other and with interrupt polling on specific cycle parities. The old `cpu.Stall(513)` + per-cycle `StallStepper` model couldn't express the interleave.
- **Decision (chippy-side, chippy#377):** port Mesen2's `ProcessPendingDma` (`Core/NES/NesCpu.cpp:325-447`). Peripherals only flag intent — `OAMDMA.Write` → `cpu.SetNeedSpriteDma(page)`, `dmcChannel.maybeRefill` → `cpu.SetNeedDmcDma()` — both set `needHalt`. On the next `CPU.read` at opcode fetch where `needHalt` is set, the drain runs: a halt cycle (dummy read at the opcode-fetch PC), then a loop on cycle parity — sprite reads on getCycle, sprite writes on putCycle, DMC reads merged in, alignment dummies inserted as needed. The nessy-side `dma/` peripheral is now just the state-signal flag; the bus-steal runs inside the CPU.
- **Consequences:** `irq_and_dma` (test 4) passes and the DMC↔OAMDMA contention is exact. The CPU owns DMA entirely — driving the DMA from outside is no longer a supported path. nessy must wire `SetDMCFetcher` so the fetched byte has somewhere to land.

## D8 — APU frame counter: non-uniform 6-substep NTSC table

- **Context:** modelling the NTSC 4-step frame counter as 4×7457 (=29828) cycles puts the half-frame tick on the wrong cycle and fails `apu_test` IRQ-timing sub-tests.
- **Decision:** port Mesen2's 6-internal-substep table (`Core/NES/APU/ApuFrameCounter.h:19`). The NTSC 4-step intervals are NON-UNIFORM and total **29830** CPU cycles per frame: `frameStepIntervalsNtsc4Step = [6]int{7456, 7458, 7457, 1, 1, 7457}`. The user-visible "step 3" spans CPU cycles 29828–29830; the half-frame tick fires at cycle 29829 (Mesen step 4), not 29828. The 5-step analogue is `[6]int{7456, 7458, 7458, 7452, 1, 7457}`. This was promoted from an earlier 4-entry form (chippy#377 → chippy#380) and ported to nessy in the same PR.
- **Consequences:** `apu_test` IRQ-flag and IRQ-timing sub-tests pass. The table is the canonical schedule for both quarter- and half-frame ticks; flipping the $4017-write delay parity without a Mesen cross-check will silently desync it.

## D9 — DMC clock semantics + `$4015` IRQ-ack rules

- **Context:** the DMC channel's clock divider and IRQ-acknowledge behavior have several real-silicon edge cases the `dmc_basics` / `dmc_rates` sub-tests pin.
- **Decision:** match Mesen exactly (`Core/NES/APU/DeltaModulationChannel.cpp:119-164`, chippy#382) — **8 fires per byte** (not 9), fire-to-fire = period (not period+1), schedule-fetch check runs every clock (not just at boundaries). `bitsRemaining` initialises to 8; reload to `period - 1` so the down-counter ticks `period` cycles between fires (Mesen `ApuTimer::Run` advances by `_timer + 1`). **A `$4015` read does NOT clear the DMC IRQ flag** — only the frame-counter IRQ is cleared on `$4015` read (`Core/NES/APU/NesApu.cpp:97-105`, chippy#381); DMC IRQ acks via `$4015` write or `$4010` bit-7 clear. `maybeRefill` silences only when the buffer is empty AND `bytesRemaining == 0`; DMC enable schedules an initial fetch when the buffer is empty with bytes pending, initialising `bufferEmpty=true, silenced=true`.
- **Consequences:** `apu_test` reaches 8/8. These four DMC rules are interdependent — `dmc_basics` test 10 pins the `$4015`-read rule specifically.

## D10 — PPU `renderingEnabled` 1-PPU-clock delay

- **Context:** the NTSC odd-frame dot-skip latch reads the render-enable state, and reading the *live* mask register puts the skip on the wrong frame, failing the last `ppu_vbl_nmi` sub-test.
- **Decision (nessy-side, chippy#375):** the odd-frame dot-skip latch at pre-render dot 339 uses `renderingEnabledDelayed` — synced at the end of each `stepDot` from the live mask — NOT the live mask. This follows Mesen's `NesPpu::UpdateState` comment: "Rendering enabled flag is apparently set with a 1 cycle delay." (`Core/NES/NesPpu.cpp:553-582`, odd-frame skip at `:946-955`.)
- **Consequences:** `ppu_vbl_nmi` reaches **10/10** — the hard accuracy gate. This invariant lives in this repo's `internal/nes/ppu`, not the chippy dep.

## Note

The invariants in D5–D10 are codified in this repo's `CLAUDE.md` under "Load-bearing invariants" (split into chippy-side and nessy-side subsections) and narrated with their Mesen source line references under "How this code came to be." `docs/context.md` carries the longer-form architecture + history dump. After the carve, the chippy-side invariants (D5–D9's CPU details) live in the `github.com/nkane/chippy` dependency pinned in `go.mod`; the nessy-side invariants (D7's `dma/` flag, D8's APU table, D10's PPU delay) live in `internal/nes`. The accuracy harness (`cmd/nessy/accuracy_test.go`) is the executable record that all of this still holds — run `go test -tags=accuracy ./cmd/nessy/...` before claiming an accuracy fix landed.
