# ADR 0001 — v0.1.0: NROM foundation, Ebiten window, and DAP-attach as a first-class feature

- **Status:** Accepted
- **Release:** v0.1.0 (2026-05-15; folds v0.1.2)
- **Theme:** Stand up a playable NROM-only NES on top of the chippy 6502 + DAP library, with source-level debugging wired in from the first frame.

## D1 — Build *on* the chippy `cpu` + `dap` packages, never reimplement the 6502

- **Context:** nessy needs a cycle-accurate 6502 and a debugger. chippy already ships a battle-tested 2A03-capable CPU core (`cpu.VariantNES` — NMOS minus decimal) and a Debug Adapter Protocol server, both library-shaped with no graphics deps.
- **Decision:** depend on `github.com/nkane/chippy` and consume its `cpu` (CPU + bus + `peripheral.Peripheral` MMIO) and `dap` (DAP server) packages. nessy owns only the NES-specific silicon (cart, PPU, joypad) and never constructs its own CPU — `cpu.NewVariant(bus, cpu.VariantNES)` is the single seam.
- **Consequences:** every CPU-core accuracy fix lands in chippy and flows in via a version bump rather than a fork. This pin is the dependency that the later carve-out (ADR 0008) cleanly splits the repos along, and the per-cycle interleave work all lives behind this same boundary.

## D2 — NROM (mapper 0) only, with a typed "unsupported" gate

- **Context:** a first playable build needs the simplest cartridge mapper; supporting the full mapper matrix up front would dilute the integration effort.
- **Decision:** ship the `internal/nes/cart` package with a `Cartridge` interface (`CPURead/CPUWrite/PPURead/PPUWrite` + `Mirroring`) and an `Open(*nes.ROM)` factory that dispatches by mapper number. Only NROM is implemented; every other mapper returns an explicit "unsupported in v0.1" error rather than failing obscurely. NROM handles 16 KiB (mirrored into `$C000-$FFFF`) and 32 KiB PRG, plus CHR-ROM and CHR-RAM variants.
- **Consequences:** the interface + factory shape is the extension point every later mapper (MMC1, UxROM, CNROM, MMC3, the VRCs, FME-7, AOROM) slots into without touching the bus wiring. The typed gate keeps unsupported-ROM behaviour honest.

## D3 — iNES / NES 2.0 loader as the first nessy-specific code

- **Context:** the cart factory needs a parsed ROM; header handling is fiddly and a frequent source of silent corruption.
- **Decision:** the `internal/nes` (`ines`) package parses iNES files (NES 2.0 extensions best-effort ignored for v0.1) into a `*ROM`, with named sentinel errors for bad magic, zero PRG banks, and truncated/malformed bank data so callers can `errors.Is()` them. The mapper byte is assembled from the flag6/flag7 high nibbles.
- **Consequences:** a stable, testable boundary between bytes-on-disk and emulated silicon. Region detection and the full NES 2.0 surface grow here in later releases without changing the parse contract.

## D4 — Ebiten game window (CGO), isolated behind a `//go:build nessy` tag

- **Context:** rendering + audio + input want a real game-loop framework (Ebiten), but Ebiten pulls in CGO and X11/GL dev headers on Linux that the chippy library's default CI runners do not carry — and at this point nessy still lives inside the chippy monorepo.
- **Decision:** put every Ebiten-dependent file (`cmd/nessy/*`) behind `//go:build nessy`, with a `!nessy` stub (`main_stub.go`) that prints build instructions so plain `go build ./...` stays green on every platform. The full build is `go build -tags=nessy ./cmd/nessy`. Wiring logic (`wiring.go`) stays untagged so it remains unit-testable everywhere. The game loop runs ~29830 CPU cycles per Ebiten `Update` (1.789773 MHz ÷ 60 fps), blits `ppu.FrameBuffer` via `screen.WritePixels` under `Layout(256, 240)`.
- **Consequences:** the CGO/graphics split this tag enforces is precisely the friction that motivates the eventual chippy ↔ nessy repo split (ADR 0008) — keeping a pure-Go 6502 library from dragging in the NES + Ebiten dep graph. The tag is a monorepo-isolation artifact that the build chain still expects post-carve, so it stays.

## D5 — DAP attach as a first-class, one-shell feature

- **Context:** source-level debugging should be available out of the box, not a separate ceremony — and the emulator and debugger are two processes that must not both drive the CPU.
- **Decision:** `cmd/nessy` opens a DAP listener on TCP port `:14785` by default (`-dap-port`, `0` disables), so `chippy -nessy ROM` launches the game and an attachable debug server from one shell. A `-wait-for-debugger` flag gates boot: the CPU is paused at the reset vector until a DAP client attaches, so a fast launcher can attach before the first instruction. Attaching hands CPU execution to the DAP server (the game loop pauses) so the two processes never double-step. The `.dbg` source map is threaded into the attach flow for source-level breakpoints.
- **Consequences:** the launcher/attach handshake is pinned by an end-to-end test (`wait_for_debugger_test.go`) that spawns a real binary, dials the listener, attaches, and asserts PC is still at the reset vector. This contract is load-bearing enough that the v0.1.2 patch is mostly hardening it: the `-wait-for-debugger` boot gate, routing `readMemory` through `MMIO.Peek` so cart bytes show, multi-path `.dbg` source resolution, and pause-on-attach ownership all land there.

## D6 — Hand-rolled ca65 demos doubling as framebuffer / audio regression tests

- **Context:** with no licensed ROMs to ship and the test ROM suite not yet wired, the integration path (iNES → cart → MMIO → CPU + PPU + joypad) needs deterministic, in-tree coverage.
- **Decision:** author small ca65 demos under `roms/demos/` (hello-bg static title screen, input-echo joypad indicators, vblank-bounce NMI-driven animation), each with a `Makefile` to rebuild the `.nes` from source. Untagged test drivers (`cmd/nessy/demo_*_test.go`) boot each ROM headlessly through `buildNES`, advance a fixed number of frames, hash the framebuffer, and compare against a pinned SHA — with multi-frame SHAs asserted distinct to catch a frozen NMI line or broken update path.
- **Consequences:** every layer of the v0.1 stack — palette/nametable/attribute/pattern fetch, the CPU reset + 2-vblank PPU warmup, the `$FFFA` NMI service path, the variant-gated `JMP self` halt heuristic, and joypad input — gets a deterministic regression gate that needs no external assets. These demos remain the in-tree smoke layer beneath the later downloaded-on-demand accuracy ROM harness.
