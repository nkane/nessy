# ADR 0007 — v0.7.0: VRC7 OPLL synth, multi-region timing, headless recorder, accuracy gate

- **Status:** Accepted
- **Release:** v0.7.0 (2026-05-25)
- **Theme:** Round out expansion audio (VRC7 FM), make timing region-aware (PAL/Dendy), and stand up the CI-friendly capture + accuracy-ROM safety net.

## D1 — VRC7 OPLL FM synthesis (mapper 85 audio)

- **Context:** the VRC7 cart shipped in v0.6 but silent (mapper banking only). Lagrange Point — the sole commercial NES VRC7 title — needs its Yamaha YM2413 (OPLL) soundtrack to actually play. The OPLL is a 2-operator FM synth, structurally unlike the 2A03's subtractive pulse/triangle/noise channels and unlike the additive VRC6/Sunsoft 5B square-wave expanders already in the tree.
- **Decision:** add `apu.VRC7Audio` — six melodic 2-operator (modulator → carrier) FM voices driven from the 15-entry fixed instrument patch ROM plus one user-programmable patch, latched through the cart's `$9010` address / `$9030` data port pair (`$00-$07` user patch, `$10-$25` F-number/block/key-on/sustain, `$30-$35` instrument + volume). Implement it as a **functional** FM synth: phase + ADSR envelope advance once per emitted sample (`Output` called at `apu.SampleRate`) using float sine FM, rather than porting the chip's exact log/exp LUT pipeline. KSL, vibrato/tremolo depth, and the precise ADSR rate tables are simplified.
- **Consequences:** Lagrange Point's soundtrack plays (`index.md` flips it from "silent" to audible). The deliberate "audible + recognisable, not cycle-exact" fidelity bar keeps the implementation small and testable (`vrc7_test.go` checks idle-silence, key-on output, and key-off decay) but means a future bit-exact OPLL pass would be a separate effort. Expansion-audio coverage is now VRC6, Sunsoft 5B (YM2149 clone), and VRC7 (OPLL) — three distinct synthesis families behind the one APU mixer (whose linear-sum approximation remains tracked for the non-linear DAC rework).

## D2 — Region-aware timing: PAL + Dendy alongside NTSC

- **Context:** the whole clock/geometry stack (CPU step budget, PPU scanline grid, APU frame-counter step length) was NTSC-hardcoded. The licensed library nessy targets includes PAL releases, and Dendy clones are a real third region. These differ in CPU clock, refresh rate, scanline count, and odd-frame behaviour.
- **Decision:** introduce `nes.Timing`, a value struct bundling the per-region constants (`CPUClockHz`, `FPS`, `CyclesPerFrame`, `ScanlinesPerFrame`, `PreRenderScanline`, `OddFrameSkip`, `QuarterFrameCycles`) with three instances: `NTSC` (2C02, 60 Hz, 1.789773 MHz, 262 lines, odd-frame dot-skip), `PAL` (2C07, 50 Hz, 1.662607 MHz, 312 lines, no dot-skip), and `Dendy` (50 Hz at a PAL-ish CPU rate but NTSC-shaped 262-line geometry). `TimingFor(tv)` maps the iNES/NES 2.0 TV-system field (`flag12`) to a `Timing`; `TVDual` and any unknown value fall back to NTSC.
- **Consequences:** PAL/Dendy carts run at their native frame cadence and frame-counter periods, with NTSC explicitly the safe default so the entire existing demo + accuracy corpus stays pinned to its known-good geometry. Constants that genuinely don't vary across regions (341 dots/scanline, vblank at scanline 241 dot 1, 240 visible lines) are documented as invariant rather than re-derived per region. The accuracy harness keeps its own NTSC-pinned 29830-cycle frame budget (the `accuracy` build tag can't see the `nessy`-tagged game const), so region selection never perturbs the Blargg gate.

## D3 — Headless deterministic recorder (`cmd/nessy-record`)

- **Context:** capturing a clip for docs/PR review meant screen-recording an Ebiten window — non-deterministic, GL-dependent, and impossible in CI. The emulator already produces a PPU framebuffer and a drained APU sample batch per frame; nothing forced those through a window to capture them.
- **Decision:** add `cmd/nessy-record`, a separate binary that drives `internal/nes` with no Ebiten/GL dependency. It steps frame-by-frame, snapshots each frame's framebuffer as a video frame and each frame's drained APU samples as the audio track, and applies a scripted JSON input timeline to the joypad. GIF output is pure stdlib (`image/gif`, with a palette built from the frames — NES is ≤64 colours, well under GIF's 256 cap); MP4 output (H.264 video + AAC audio) shells out to `ffmpeg`, the dependency the smoke job already installs.
- **Consequences:** the same ROM + script produce a byte-identical recording every run, so clips become reliable PR smoke artifacts with no timing flake. Capturing video + audio from the same `stepFrame` keeps them in sync by construction. The recorder shares `internal/nes` with the game and WASM binaries, so it exercises the real emulator core, not a stub.

## D4 — Blargg/nesdev accuracy suite + `knownFail` gate pattern

- **Context:** the cycle-precision CPU/PPU/APU work (carried over from the chippy monorepo) needed a standing regression net, and the test-ROM corpus is GPL — it can't be vendored. Many ROMs expose *real, tracked* accuracy gaps; a strict pass/fail gate would either be perpetually red or tempt deleting the failing ROMs.
- **Decision:** add an `accuracy`-build-tagged suite (`cmd/nessy/accuracy_test.go`) that runs each ROM through the full iNES → cart → MMIO → CPU+PPU+APU integration and reads Blargg's `$6000` status protocol (`$6001-$6003` magic gates trusting `$6000`; `$6004+` is the result text). ROMs are env-overridable, otherwise downloaded + SHA-pinned into a local cache (downloads retry transient 5xx/429 with linear backoff so a flaky GitHub-raw 504 can't red the gate). Each fixture carries an optional `knownFail` string: a ROM that fails is **skipped with its tracked-gap note** instead of failing the build, while any regression in a currently-passing ROM is a hard failure.
- **Consequences:** the passing suite (ppu_vbl_nmi 10/10, instr_timing, cpu_interrupts_v2 5/5, apu_test 8/8, instr_misc 4/4, instr_test-v5_official 16/16, oam_read, mmc3_test 1/2/3/5) stays green in CI while open gaps (oam_stress, ppu_open_bus, mmc3_test 4/6, the visual-only and init-hanging ROMs) are documented inline against their tracking issues rather than hidden. Clearing a `knownFail` is the explicit signal that a gap closed. This harness is the safety net every subsequent accuracy change is measured against — the suite and its Mesen2 reference deepen in ADR 0008.
