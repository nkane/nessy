# ADR 0003 — v0.3.0: full 5-channel APU, frame counter, non-linear DAC mixer, MMC1

- **Status:** Accepted
- **Release:** v0.3.0 (2026-05-24)
- **Theme:** Complete the 2A03 audio path — triangle / noise / DMC join the two pulse channels under a 4-step/5-step frame counter and a non-linear DAC mixer — and add the first serial-shift-register mapper, MMC1.

## D1 — Triangle and noise channels on the existing `cpu.Ticker` clock

- **Context:** v0.2 shipped only the two pulse channels, the $4017 frame counter, and a 44.1 kHz int16 sample ring; the triangle ($4008-$400B) and noise ($400C-$400F) register windows accepted writes but emitted silence. The APU already advances per CPU cycle as a `cpu.Ticker` (the bus-ticker hook from ADR 0002 D4).
- **Decision:** add `triangleChannel` and `noiseChannel` as siblings of `pulseChannel`, each ticked from `APU.stepCPU`. The pulse and noise period timers clock every *other* CPU cycle (`alternateTick`); the triangle period timer clocks *every* CPU cycle, because its 32-step sequencer needs the higher rate to reach audible frequencies. Quarter-frame ticks drive the triangle's linear counter and the noise envelope; half-frame ticks drive both length counters.
- **Consequences:** the per-cycle ticker established in ADR 0002 carries the whole audio stack at zero extra plumbing cost. The channel set is feature-complete enough to render the triangle / noise demos added in the same release (the `roms/demos` regression ROMs).

## D2 — Frame counter: 4-step / 5-step modes with the delayed $4017 side effects

- **Context:** $4017 selects the frame-counter mode (bit 7: 0 = 4-step / 240 Hz IRQ, 1 = 5-step / no IRQ) and the IRQ-inhibit flag (bit 6). On real silicon the mode latch, counter reset, and the 5-step "immediate" quarter+half-frame tick do **not** take effect at the write — they are delayed 3 or 4 CPU cycles depending on whether the write lands between or during an APU cycle. Only the IRQ-inhibit flag clear is immediate.
- **Decision:** model the side-effect split explicitly: `SetFrameCounter` applies the inhibit/flag-clear immediately, then arms `frameResetDelay` (3 or 4 by cycle parity) with the latched `$4017` byte; `stepCPU` counts the delay down and calls `applyFrameCounterReset` when it expires. The $4017 write itself arrives forwarded from `joypad.Port` so the two peripherals don't fight over the shared register address.
- **Consequences:** the delayed-reset model is load-bearing for interrupt timing — `cpu_interrupts_v2` tests 3-5 fail without it because the frame IRQ asserts a few cycles early and the calibration trial services an IRQ-hijack-NMI instead of a plain NMI. The frame counter's named-source IRQ (`"apu-frame"`) rides the multi-source IRQ surface wired in this same release.

## D3 — Non-uniform 6-sub-step frame-counter interval table (Mesen-aligned)

- **Context:** the naive frame counter reloads a uniform 7457 CPU cycles per quarter-frame step, summing to 4 × 7457 = 29828 cycles per 4-step cycle. That is 2 cycles short of real silicon and fires the half-frame tick one cycle too early for Blargg `apu_test` length-timing's polling window.
- **Decision:** expand the 4-step sequence into 6 internal sub-steps with a non-uniform interval table, ported from Mesen2 `ApuFrameCounter.h`: `frameStepIntervalsNtsc4Step = [6]int{7456, 7458, 7457, 1, 1, 7457}`, summing to **29830** cycles per cycle. The user-visible "step 3" spans 3 CPU cycles (29828, 29829, 29830); the IRQ assertion window is 3 cycles wide and the half-frame tick fires at cycle 29829, not 29828. The 5-step analogue is `[6]int{7456, 7458, 7458, 7452, 1, 7457}` (sum 37282). Non-NTSC regions keep the uniform `quarterFrameCycles` reload (set by `SetRegion`).
- **Consequences:** this is a permanent invariant recorded in `CLAUDE.md` — the non-uniform totals (29830 / 37282) and the 6-entry tables must not be "simplified" back to 4 uniform steps. It clears the `apu_test` length-timing / IRQ-timing sub-tests. The same Mesen table was ported into chippy when chippy + nessy were one repo (chippy#380).

## D4 — DMC channel reusing the CPU's DMA-stall hook; Mesen-exact clocking

- **Context:** the delta-modulation channel ($4010-$4013) plays 1-bit delta samples fetched from CPU memory. Two side effects set it apart from the other four channels: the sample-byte fetch steals CPU cycles (a DMA stall), and sample exhaustion can assert an IRQ. The CPU-side DMA-stall machinery already existed for OAMDMA.
- **Decision:** the DMC requests a refill via `fetchPending`; the actual byte read is drained inside the CPU's `ProcessPendingDma` loop, which issues `bus.Read(APU.GetDmcReadAddress())` and hands the byte back through `APU.SetDmcReadBuffer`. The DMC clock matches Mesen exactly: 8 fires per byte (not 9), fire-to-fire interval = period (not period+1), `bitsRemaining` initialises to 8, and the period reloads to `period - 1` so the down-counter ticks `period` cycles between fires. The DMC IRQ rides its own named source (`"apu-dmc"`).
- **Consequences:** reusing the OAMDMA stall hook avoided a second bespoke DMA path and set up the later per-cycle DMC/OAMDMA interleave (the `ProcessPendingDma` cycle-parity merge that landed across chippy#377 and the carve-out — ADR 0004). Several DMC silicon details are pinned in `CLAUDE.md`: `$4015` read does **not** clear the DMC IRQ flag (only the frame IRQ); `$4015` write or `$4010` bit-7 clear acks it; `maybeRefill` silences only when the buffer is empty **and** bytes-remaining is zero; and DMC enable schedules an initial fetch. Blargg `apu_test` `dmc_basics` / `dmc_rates` pin these.

## D5 — Non-linear DAC mixer replacing v0.2's linear sum

- **Context:** v0.2 combined the channels with a linear approximation — roughly `pulse1 + pulse2 + (...)` scaled to int16. Real silicon mixes through two non-linear DACs (a pulse DAC and a triangle/noise/DMC DAC) per the nesdev "APU Mixer" page, so a linear sum gets the relative channel levels audibly wrong.
- **Decision:** implement the nesdev non-linear formulas in `mixer.go`. The pulse term depends only on `p1 + p2` (31 distinct values), so it is precomputed once into `pulseTable`; the tnd term has too many input combinations (16 × 16 × 128) to make a full LUT worthwhile, so it is evaluated inline per sample. `mixSample` returns a float in roughly [0, 1.0] that `emitSample` scales to int16.
- **Consequences:** this *replaced* the v0.2 linear mixer outright; the linear approximation is gone. Cartridge expansion-audio chips (Sunsoft 5B, VRC6, VRC7) are summed onto the post-DAC sample rather than fed through `mixSample`, because on real hardware those chips live on the cart and are mixed with console audio on the cart's analog out — outside the 2A03's DAC. The per-chip scale constants are tuned for plausible relative level, not silicon-exact balance; refining the expansion-audio balance and the int16 headroom is deferred to the v1.0 audio epic (#5, the non-linear DAC mixer line item) and later ADRs.

## D6 — MMC1: the first serial-shift-register mapper (mapper 1)

- **Context:** the mappers supported through v0.2 (NROM, plus parallel-register families) latch a bank value in a single write to a parallel register. MMC1 — Nintendo's first MMC ASIC, behind Zelda 1, Final Fantasy, Metroid, Castlevania II, Dragon Warrior — is fundamentally different: it assembles a 5-bit value one bit per write.
- **Decision:** model the serial protocol directly. Each write to $8000-$FFFF shifts bit 0 into a 5-bit accumulator; on the fifth write the assembled value commits to one of four internal registers selected by destination-address bits 13-14 (control / CHR bank 0 / CHR bank 1 / PRG bank). A write with bit 7 set resets the shift register and ORs the PRG-mode bits to 3. The cart powers up with PRG mode 3 latched (last bank fixed at $C000). Mirroring derived from control bits 0-1 is cached so `Mirroring()` stays O(1); battery-backed PRG-RAM at $6000-$7FFF is exposed via `PRGRAM()` for save/restore.
- **Consequences:** the serial-shift model is the template later serial mappers reuse. The MMC1 "consecutive-cycle" silicon bug — back-to-back writes from RMW opcodes are ignored — was scoped out of v0.3 and noted in the code as a known gap. Battery support routes saves to a sibling `.sav`, completing the save-state semantics user-facing docs describe.
