# ADR 0006 — v0.6.0: VRC6 expansion audio, VRC7 cart shell, WASM playground, DMA contention

- **Status:** Accepted
- **Release:** v0.6.0 (2026-05-24)
- **Theme:** Broaden Konami expansion-audio mapper coverage, ship an in-browser playground on the shared core, and start modelling single-DMA-controller bus contention.

## D1 — VRC6 3-channel expansion audio (mappers 24 / 26)

- **Context:** Castlevania III (JP) and other Konami VRC6 titles route a dedicated audio chip — two pulse channels plus a sawtooth — through the cart on top of a VRC4-shape PRG/CHR/IRQ surface. The base 2A03 mixer had no path for cart-supplied channels.
- **Decision:** add `cart.VRC6` (`internal/nes/cart/vrc6.go`) covering both mapper 24 (VRC6a, A0/A1 routing) and mapper 26 (VRC6b, A1/A0 swapped) via a `subBits()` variant selector. Audio registers ($9000-$9002 pulse 1, $A000-$A002 pulse 2, $B000-$B002 sawtooth) forward through a `cart.VRC6AudioSink` hook. `cmd/nessy` wires an `apu.VRC6Audio` into the mixer at construction time via the `SetAudioSink` type-assertion probe — the same shape used to wire Sunsoft 5B for FME-7.
- **Consequences:** establishes the cart-audio Sink pattern as the seam for every expansion chip; the APU mixer gains expansion inputs without the cart package depending on `apu`. Mixer stays a linear sum + scale (non-linear DAC deferred to the v1.0 epic).

## D2 — VRC7 cart shell only; OPLL FM synth deferred (mapper 85)

- **Context:** VRC7 (Lagrange Point) packages a Yamaha YM2413 (OPLL) FM synthesizer alongside a VRC4-shape PRG/CHR/IRQ surface. The OPLL is large enough that bundling it with the mapper would have made one oversized PR.
- **Decision:** ship the cart side only in this release (`internal/nes/cart/vrc7.go`): PRG/CHR banking, VRC4-style IRQ (CPU + scanline modes, 8-bit counter), mirroring/WRAM control, and the audio-port address routing ($9010 register select / $9030 data write). Audio-register writes forward through a `cart.VRC7AudioSink` so a future `apu.OPLL` drops in without touching the cart.
- **Consequences:** mapper-85 ROMs load and run, with the FM voices silent. The OPLL FM synth completes in **ADR 0007**, sliding in behind the already-wired sink with no cart-side changes.

## D3 — WASM playground on the shared `internal/nes` backend

- **Context:** an in-browser demo lowers the barrier to trying nessy, but the desktop binary pulls in Ebiten's CGO/GL desktop path.
- **Decision:** add `cmd/nessy-wasm` (`//go:build js && wasm`), built with `GOOS=js GOARCH=wasm`. It reuses the exact `internal/nes` backend (cart / ppu / apu / dma / joypad) and Ebiten's js/wasm path (which drives `requestAnimationFrame` internally), with no desktop windowing. A default demo ROM is embedded so the page boots into live output; a JS-side `nessy.loadROM(Uint8Array)` swaps the running bus. The HTML/JS shell lives in `web/nessy`.
- **Consequences:** three front-ends (`cmd/nessy`, `cmd/nessy-record`, `cmd/nessy-wasm`) now share one emulator core. The wasm bus wiring is duplicated from `cmd/nessy/wiring.go` rather than imported, since the two `main` packages can't share types without a refactor. Audio is not yet wired on the wasm path (the js/wasm audio context needs a user-gesture unlock); samples are drained each frame to keep the ring buffer from filling. Playground UX hardening (File-API ROM picker, audio unlock) is tracked under the v1.0 epic.

## D4 — DMC / OAMDMA single-controller bus contention

- **Context:** real silicon has one DMA controller. When a DMC sample fetch fires inside an in-flight OAMDMA window, the DMC fetch must wait for an OAMDMA byte slot and pays an alignment penalty.
- **Decision:** add a `PendingStall() int` accessor on the CPU and extend the `apu.DMCStaller` interface with it. `dmc.maybeRefill` bumps the normal 4-cycle DMC stall to 6 when the staller reports non-zero pending DMA debt. The test baselines a non-contended run, then re-runs with `pending=100` and asserts the per-fetch stall went 4 → 6.
- **Consequences:** directionally correct and bounded — most shipping ROMs sit outside the contention envelope, so the approximation is safe. A sample-perfect model (nesdev's `dmc_dma_during_read`) needs per-cycle DMA accounting nessy did not yet have at this release. That accounting arrives with the Mesen2 `ProcessPendingDma` port, which subsumes this approximation — see **ADR 0008**.

## D5 — Daily-driver CLI: recent-ROMs list + controller rebinding

- **Context:** launching the same ROMs repeatedly and remapping keys are quality-of-life gaps for day-to-day use.
- **Decision:** add a recent-ROMs list at `~/.nessy/recent` (up to 5 absolute paths, newest first, deduped). `nessy` with no args prints the numbered list to stderr and exits; `nessy N` (1..5) opens the Nth slot; a positional path still opens verbatim; each successful boot records its absolute path. Add controller rebinding at `~/.nessy/controller.json` mapping per-player NES button names to Ebiten key names (`{"p1": {"A": "Space", ...}}`), with tolerant identifier normalisation (case / whitespace / dashes / underscores) overlaying the package-level `keyMap`. A missing file is silent; a malformed file warns and keeps defaults.
- **Consequences:** `~/.nessy/` becomes the home for nessy user config alongside the existing save-state directory. Both features are best-effort — persistence errors warn but never fail the boot.
